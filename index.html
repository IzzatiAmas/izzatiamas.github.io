Lab Work

Progressive Lab Journey

1

Lab 1 — User Interface

Jetpack Compose · Layouts
▶
https://youtu.be/wLxS0ulHZGc

In Lab 1, I built the foundational UI for a Plant Tracker app using Jetpack Compose, practising layouts like Column, Row, and Box to structure a home screen with a header, summary card, and static plant list. I learned how to define a custom brand colour palette and apply it consistently across composables. This lab helped me understand the declarative UI paradigm in Compose and how to build a clean, readable screen without XML.

2

Lab 2 — Interaction & State

remember · mutableStateOf · LazyColumn
▶
YouTube link to be added

Lab 2 introduced state management with remember and mutableStateOf, allowing me to build a dynamic plant list where users can add new plants through an OutlinedTextField and a button. I implemented a LazyColumn to efficiently render the growing list and updated the summary card count reactively. This taught me how user interactions drive UI recomposition in Compose.

3

Lab 3 — Material Design 3

Material Theme · Animations · Color.kt
▶
YouTube link to be added

In Lab 3, I applied Material Design 3 theming by defining a full custom colour scheme through Color.kt and Theme.kt, and added expand/collapse animations to plant cards using animateContentSize with a spring animation spec. I also wrapped the app in a proper MaterialTheme composable, ensuring consistent token-based styling throughout. This lab showed me how Material 3's theming system makes it easy to create a polished, branded look.

4

Lab 4 — Navigation

NavHost · NavController · ViewModel
▶
YouTube link to be added

Lab 4 introduced multi-screen navigation using NavHost and NavController, with screens for Login, Home, Add Plant, Detail, Profile, and Settings, and a BottomNavigationBar that hides on the login screen. I also implemented a simple AuthViewModel to manage login state and a PlantViewModel with an in-memory plant list shared across screens. This lab made me appreciate how navigation and state management work together in a real app architecture.

5

Lab 5 — Room Database

Room · @Entity · @Dao · Flow · CRUD
▶
YouTube link to be added

Lab 5 extended the navigation app with Room database persistence, defining a Plant entity with @Entity, a PlantDao with @Insert, @Delete, @Update, and a Flow-based getAllPlants() query, and a PlantRepository to abstract database access. I added a full Update screen and wired the PlantViewModel to collect the Room Flow as Compose state, so the UI updates automatically on any change. This lab gave me hands-on experience with the full CRUD lifecycle using Room and Kotlin Coroutines.

Projects

BloomIt — Built in Two Phases

Project 01 · Lab 4 + Working Features

BloomIt 🌸

Problem Statement Plant enthusiasts often forget care routines and lose track of their collection. This app gives users a personal garden journal where they can log, explore, and manage their plants with navigation across dedicated screens.
▶
YouTube VSR link to be added
Lessons Learned Building Project 1 taught me how to integrate multiple screens into a cohesive app with proper back-stack management, including passing data (plant name) as a navigation argument to the Detail screen. I learned to separate data concerns by using a PlantViewModel with a shared mutable state list and an ExploreData file for curated flower content. Project 02 · Lab 5 + API + Sensors + Cloud

BloomIt Pro 🌿

Problem Statement Plant care is often guesswork; users need real-time environmental data and smart identification tools to care for plants properly. This app combines Room-persisted plant journaling with live weather data, AI-powered flower identification via camera, and a Firebase Firestore community garden board.
▶
YouTube VSR link to be added
Lessons Learned Project 2 pushed me to integrate multiple external systems — the Plant.id API for image-based flower identification, the OpenWeatherMap API triggered by GPS coordinates, and Firebase Firestore with real-time listeners for the community screen. I learned to manage multiple StateFlow-based ViewModels running network calls on Dispatchers.IO inside viewModelScope, and handle Android camera access via FileProvider. Combining local Room storage with cloud Firestore also taught me to think carefully about which data belongs locally and which should be shared.

Reflection

Learning Journey

This course took me from building a single static screen to a full multi-feature Android app with cloud integration, covering the complete Jetpack Compose stack in progression. Each lab built directly on the previous one — UI → state → theming → navigation → persistence — which made each new concept feel natural rather than overwhelming. The project phase was where everything clicked: having to connect a camera, a REST API, GPS sensors, and Firebase Firestore in one app forced me to think architecturally about ViewModels, coroutines, and data flow. I leave this course with a real working app and a solid foundation for professional Android development.
